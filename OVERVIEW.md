## Purpose

This repository is a local-first, git-versioned funding strategy system for Windsor Forest Brass Academy (WFBA).

Its purpose is to help WFBA identify, qualify, prioritise, and pursue suitable funding opportunities in a systematic, inspectable way. The system should reduce ad hoc grant searching, preserve institutional memory, and focus limited capacity on the highest-value prospects for the organisation's mission, programmes, and stage of development.

This is not just a grant-finder. It is a funding pipeline operating system covering:
- source monitoring
- funder and opportunity qualification
- project-to-funder matching
- evidence readiness
- application planning
- relationship and stewardship tracking

## Canonical opportunity states

Every opportunity must be assigned exactly one of the following states. These are the only valid values.

| State                   | Meaning                                                                                      |
|-------------------------|----------------------------------------------------------------------------------------------|
| `apply_now`             | All eligibility gates passed; prioritise for immediate action                                |
| `apply_when_registered` | Requires confirmed charity number; revisit the moment registration is confirmed              |
| `monitor`               | Open or recurring round; not yet actionable but worth watching                               |
| `relationship_build`    | No open round; cultivate contact for future access                                           |
| `parked`                | One or more eligibility gates unclear; specific blocker must be recorded; revisit trigger defined |
| `rejected`              | Definitively ineligible or deprioritised; reason must be recorded                           |

Rules:
- Every opportunity file must include a `state` field and a `state_rationale` field.
- Transitioning an opportunity from any state to another requires a dated note explaining what changed.
- `parked` is not a dumping ground; a revisit trigger must always be defined.
- When charity registration is confirmed, run a full sweep of all `apply_when_registered` opportunities immediately.

## How the system should think

### Data quality rules
1. Record a `last_verified_date` for every funder rule, eligibility statement, and deadline.
2. Keep provenance for every funder rule, deadline, and eligibility statement.
3. Distinguish clearly between fact, judgement, and task.
4. Prefer verified organisational facts over guessed detail.
5. Record whether evidence is individual-level, aggregate, anecdotal, or inferred.
6. Do not overclaim on deprivation, protected characteristics, or partnership scope unless supported by confirmed data.

### Search and qualification rules
1. Treat eligibility as a gating check, not merely a soft score.
2. Use the canonical opportunity states (above); do not invent new ones.
3. Allow `parked` or `rejected` as valid outcomes — not every prospect is worth pursuing.
4. Focus on opportunities that match actual programmes WFBA can plausibly deliver.
5. Prefer local and regional plus music, youth, and community funders first, unless strong evidence suggests otherwise.
6. Include government, council, and public schemes as first-class targets alongside trusts.
7. Highlight evidence gaps early rather than at application time.
8. Deprioritise small restricted grants with reporting burdens disproportionate to grant value.
9. Avoid creating busywork or low-value churn; small numbers of high-quality prospects beat noisy longlists.

### Output and behaviour rules
1. Preserve inspectability: plain text, readable notes, explicit rationale.
2. Separate facts, judgements, and actions clearly in any output.
3. Reuse application assets across similar funders wherever possible.
4. The agent should be sceptical of its own inferences and conservative with assumptions.

## Information architecture

This repository should be organised around the following core object types.

### Directory structure
```
organisation/         # authoritative facts about WFBA
programmes/           # fundable programme definitions
funder_sources/       # places monitored for new opportunities
funders/              # organisations that may run relevant schemes
opportunities/        # specific funds, rounds, or grant programmes
applications/         # actual or planned submissions
evidence/             # policies, budgets, partner letters, case studies, etc.
relationships/        # warm contacts, stewardship notes, prior interactions
tasks/                # discrete next actions with owners and due dates
```

### File format convention
- Each object is a single `.md` file with YAML frontmatter, followed by free-text notes.
- File names: lowercase, hyphens, no spaces. Example: `funders/arts-council-england.md`
- **Templates in `_templates/` are mandatory starting points, not suggestions.** Do not create object files from scratch. Do not invent new field names; extend the template only if genuinely necessary, and document the addition.
- Fields marked `# REQUIRED` in templates must never be omitted, left as the placeholder value, or set to null. A record with missing required fields is incomplete and must not be acted on.
- Partnership or association claims must include a `claim_status` field: `confirmed` | `inferred` | `speculative`. Do not use `inferred` or `speculative` claims in funding applications without first confirming them.

### Mandatory fields by object type
The minimum fields that must be present and non-null before a record is considered valid:

**opportunity:** `id`, `name`, `funder`, `funder_type`, `state`, `state_rationale`, `last_updated`, `last_verified_date`, `provenance`, `geography`, `charity_registration_required`, `eligibility_summary`, all six `gate_` fields, `next_action`

**programme:** `name`, `status`, `last_updated`, `last_verified_date`, `description`, `delivery_model`, `delivery_readiness`, `delivery_readiness_rationale`, `critical_prerequisites` (at least one entry if status is `planned`), `target_beneficiaries`, `geography`, `primary_outcomes`

**funder:** `name`, `funder_type`, `last_updated`, `last_verified_date`, `source_url`, `geography_scope`, `primary_focus`, `charity_registration_required`, `relationship_status`

**evidence:** `id`, `name`, `evidence_type`, `last_updated`, `last_verified_date`, `author_or_source`, `claim_status`, `supports_programmes`, `strength`, `strength_rationale`

**organisation:** `name`, `charity_registered`, `last_updated`, `last_verified_date`, `primary_delivery_area`

### Stale record rules
Records go stale. The following thresholds trigger a mandatory recheck before any action is taken:

| Record type                               | Recheck trigger                                            |
|-------------------------------------------|------------------------------------------------------------|
| Opportunity deadline                      | Not verified within 30 days                                |
| Recurring funder scheme                   | Not verified within 90 days                                |
| `parked` opportunity                      | Must have `review_date` defined; absent = invalid record   |
| `monitor` opportunity                     | Not reviewed within 60 days                                |
| Organisation facts (`organisation/wfba.md`) | Not verified within 6 months                             |

A recheck means: return to the original source, confirm the information is still current, and update `last_verified_date`. If the information has changed, update the record and log the change in the state history section.

**1. organisation**
Authoritative facts about WFBA: legal form, registration status, income, governance, geography, contacts.

**2. programmes**

## Qualification model

The system should use a two-stage decision process.

### Stage 1 – qualification gates
Check in order:
1. legal-form fit
2. charity-registration requirement
3. geography fit
4. beneficiary fit
5. programme fit
6. funding-size fit
7. open or closed status
8. deadline realism
9. evidence sufficiency

Any likely **no** on gates 1–3 should move the prospect to `rejected` immediately unless new information appears. Uncertain gates should move the prospect to `parked` with the specific blocking gate noted.

### Stage 2 – priority scoring
Only `apply_now` and `apply_when_registered` prospects should be prioritised and scored.

Suggested scoring dimensions:
- mission_fit
- programme_fit
- evidence_readiness
- likelihood_of_success
- grant_size_fit
- relationship_value
- repeat_funding_potential
- urgency
- application_burden
- restriction_cost
- confidence

The score is heuristic only. Written rationale matters as much as the number.

## Bootstrapping guide

The repository is currently empty. The following sequence builds a working system in priority order.

### Step 1 — fill in ASSUMPTIONS.md
Open `ASSUMPTIONS.md` (created alongside this file) and complete every blank field. Do not proceed with funder research until legal form, registration status, income band, and geography are confirmed.

### Step 2 — create organisation.md
Write `organisation/wfba.md` with confirmed organisational facts. This becomes the single authoritative reference for all applications.

### Step 3 — write programme briefs
For each Priority 1–3 programme, create a file in `programmes/`. Each brief should include: objective, target beneficiaries, delivery model, budget estimate, outcomes, and evidence available.

### Step 4 — build evidence inventory
List existing evidence assets in `evidence/`. Even minimal items (safeguarding policy link, DBS confirmation, bank account readiness) are worth recording now.

### Step 5 — begin funder prospecting
Start with Priority 1 (Year 4 school teaching) given the September 2026 urgency. Search local authority, education, and music education funds first. Log each prospect immediately as an opportunity file with a state and state_rationale.

### Step 6 — run a first shortlist review
After 10–20 prospects are logged, review states. The goal is a shortlist of 3–5 `apply_now` prospects with full qualification rationale and a clear next action for each.
